# Forensic Lens - Technical Documentation

## Overview

Forensic Lens is a real-time fraud detection system that combines machine learning, graph analysis, and LLM-powered investigation in an immersive 3D visualization interface.

## 3D Graph Visualization

### ForceGraph3D Configuration

```typescript
ForceGraph3D {
  nodeThreeObject: Custom mesh rendering
  linkDirectionalParticles: Animated flow indicators
  controlType: "orbit" for camera control
  cooldownTicks: 100 for settling
  autoRotate: Optional rotation for engagement
}
```

### Node Rendering

Each node is rendered as a custom Three.js Group containing:

1. **Main Sphere**: SphereGeometry with MeshPhongMaterial
   - Size varies by risk level (8-15 units)
   - Color based on risk score
   - Emissive glow for high-risk nodes

2. **Ring Geometry** (for cycle nodes):
   - Double ring effect for fraud ring visibility
   - Red color (#ef4444) with transparency
   - Rotation for 3D orientation

3. **Text Sprite**:
   - Canvas-based texture rendering
   - Label above each node
   - Color-coded by node type

### Risk-Based Node Sizing

| Risk Level | Size | Color | Glow |
|------------|------|-------|------|
| Low (0-25%) | 8 | #22c55e | None |
| Medium (25-50%) | 10 | #eab308 | Subtle |
| High (50-75%) | 12 | #f97316 | Moderate |
| Critical (75%+) | 15 | #ef4444 | Strong |

### Link Rendering

| Type | Color | Width | Particles |
|------|-------|-------|-----------|
| Normal | rgba(255,255,255,0.25) | 2.5 | None |
| Cycle | rgba(239,68,68,0.8) | 5 | Directional |

## Machine Learning

### XGBoost Model

- **File**: `Model/model.pkl`
- **Features**: 16 columns
- **Algorithm**: Gradient Boosting (XGBClassifier)
- **Task**: Binary classification (fraud/legitimate)

#### Feature Engineering

| Feature | Description |
|---------|-------------|
| orig_diff | oldbalanceOrg - newbalanceOrig |
| dest_diff | newbalanceDest - oldbalanceDest |
| orig_zero | Boolean: sender had zero balance |
| dest_zero | Boolean: receiver had zero balance |
| isHighRiskType | Boolean: CASH_OUT or TRANSFER |
| type_* | One-hot encoded transaction types |

### Hybrid Scoring Formula

```
final_probability = ml_probability + graph_boosts + amount_boost
```

#### Graph Boosts

| Condition | Boost |
|-----------|-------|
| Degree > 3 | +5% |
| Degree > 5 | +10% |
| Clustering > 0.4 | +10% |
| Cycle Detected | +30% |
| Money Mule Chain | +35% |
| Fan-In Pattern | +25% |

#### Amount Thresholds

| Amount | Boost |
|--------|-------|
| > $250K | +10% |
| > $1M | +20% |
| > $10M | +40% |

#### Fraud Threshold

Score ≥ 70% triggers high-risk alert (blocked)

## Graph Analysis (NetworkX)

### Algorithms Used

1. **Cycle Detection** (`nx.simple_cycles`)
   - Identifies circular fund flows
   - Used for fraud ring detection
   - Returns list of cycles in the graph

2. **Clustering Coefficient** (`nx.clustering`)
   - Measures local density
   - High clustering = fraud ring pattern
   - Calculated on undirected graph

3. **PageRank** (`nx.pagerank`)
   - Identifies central/important nodes
   - Used for hub detection
   - Alpha: 0.85 (damping factor)

4. **Risk Propagation**
   - Iterative spreading of risk scores
   - 70% current + 30% neighbor average
   - 2 iterations per transaction

### Money Mule Detection

Patterns detected:
- **Chain Pattern**: Path length > 2 between sender/receiver
- **Fan-In Sink**: Multiple nodes sending to single receiver
- **Cycle Ring**: Circular flow between accounts

## Database (SQLite)

### Schema

```sql
CREATE TABLE users (
  id TEXT PRIMARY KEY,          -- Hex ID (0x...)
  username TEXT NOT NULL,
  risk_score REAL DEFAULT 50.0,
  user_type TEXT DEFAULT 'User',
  created_at TEXT,
  updated_at TEXT
);

CREATE TABLE transactions (
  id TEXT PRIMARY KEY,
  sender_id TEXT NOT NULL,
  receiver_id TEXT NOT NULL,
  amount REAL NOT NULL,
  transaction_type TEXT,
  timestamp TEXT,
  is_flagged INTEGER DEFAULT 0,
  fraud_probability REAL,
  FOREIGN KEY (sender_id) REFERENCES users(id),
  FOREIGN KEY (receiver_id) REFERENCES users(id)
);
```

## Deployment

### Railway (Recommended)

1. Connect GitHub repo
2. Add environment variable: `OPENROUTER_API_KEY`
3. Railway auto-builds frontend + deploys backend
4. Single port serves both API and frontend

### Docker

```dockerfile
FROM python:3.11-slim
WORKDIR /app
RUN pip install -r requirements.txt
COPY backend/ ./app/
COPY Model/ ./Model/
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

### Environment Variables

| Variable | Required | Description |
|----------|----------|-------------|
| OPENROUTER_API_KEY | Yes | For LLM analysis |
| PORT | No | Railway port (auto-set) |
| DATABASE_PATH | No | SQLite file location |

## Performance Considerations

### Frontend

- **3D Graph**: Uses GPU acceleration via Three.js
- **Polling Interval**: 12 seconds for graph refresh
- **Query Caching**: TanStack Query with stale time
- **Code Splitting**: Vite handles automatically

### Backend

- **Database**: SQLite with indexes on sender/receiver
- **Background Tasks**: Async data fetching from CoinGecko
- **Model Loading**: Singleton pattern (loaded once at startup)

### Graph Scaling

Current implementation suitable for:
- Up to ~1000 nodes
- Real-time updates with 5s polling
- Single-server deployment

For larger scale, consider:
- Neo4j for graph storage
- Redis for caching
- WebSocket for real-time updates
